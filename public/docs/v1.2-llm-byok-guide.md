# APA Coach v1.2 BYOK LLM 使用说明

## 1. 这是什么

v1.2 增加了可选的 LLM 增强反馈。它不是默认开启功能，也不是服务器托管的 AI 服务。

当前实现是 BYOK，也就是 bring your own key：使用者自己提供 OpenAI-compatible API 的 `base URL`、`API key` 和 `model`。

## 2. 它会不会保存学生文本

不会保存到本项目。

但是，当使用者点击 `Enhance with my LLM` 时，当前 Check 文本会从浏览器发送到使用者配置的 LLM provider。也就是说：

- 本项目不保存学生文本。
- GitHub Pages 不保存学生文本。
- 浏览器会把文本发送给你配置的第三方或本地兼容服务。
- 该 provider 如何处理数据，取决于 provider 自己的隐私政策和账号设置。

## 3. API key 保存在哪里

API key 存在浏览器的 `sessionStorage`。

这意味着：

- 刷新页面后仍可保留。
- 关闭当前标签页或浏览器会话后通常会清除。
- 不会写入 GitHub 仓库。
- 不会写入项目服务器，因为 v1.2 没有服务器。

但它仍然是浏览器端保存，不适合把个人 key 交给学生长期使用。

## 4. 为什么不建议公开学生使用自己的 key

因为 v1.2 仍是静态网页，没有安全后端代理。

如果学生输入自己的 API key：

- key 会出现在学生自己的浏览器环境中。
- 费用由 key 所属账号承担。
- provider 可能阻止浏览器直接请求。
- 课程方难以统一管理费用、隐私、速率限制和日志。

更适合的使用方式是：教授或项目负责人在受控环境中做演示和评审。

## 5. CORS 失败是什么意思

有些 LLM provider 不允许网页浏览器直接调用 API。这时页面可能显示 CORS 或 network error。

这不是 APA Coach 的 citation 规则错误，而是浏览器安全限制。

可选解决方案：

- 使用允许 browser request 的 OpenAI-compatible provider。
- 使用你自己控制的兼容代理。
- 后续版本改成安全后端代理，把 key 放在服务端环境变量中。

v1.2 不内置代理服务。

## 6. 当前 LLM 反馈边界

LLM 反馈只作为 tutoring support：

- 不验证真实 DOI、URL、作者、年份或来源真实性。
- 不保证最终 APA 格式完全正确。
- 不替代 instructor policy。
- 不自动把 MLA/Chicago 转成可靠 APA。
- 不覆盖 rule-based diagnosis。

如果 LLM 输出格式不符合预期，页面会显示安全 fallback，而不是直接展示未校验内容。

## 7. 推荐评审方式

建议教授评审时按这个顺序：

1. 先使用 rule-based Check，不配置 LLM。
2. 粘贴 MLA 示例，确认页面能识别“更像 MLA 而不是 APA”。
3. 阅读隐私提示。
4. 如果要测试 LLM，使用自己的测试 key 和低风险示例文本。
5. 确认课程政策是否允许学生把 citation 文本发送给外部 provider。

在课程政策、隐私说明、费用控制和 provider 设置没有确认前，不建议进入公开学生 pilot。
